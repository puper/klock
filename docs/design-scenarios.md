# klock 设计与场景分析

本文面向实现与运维，目标是回答四个问题：

1. 系统设计目标是什么
2. 当前实现做了哪些权衡
3. 各类真实场景下系统行为如何
4. 现在仍有哪些问题与使用注意事项

## 1. 设计目标

### 1.1 功能目标

1. 提供两级层级锁：`L1 -> L2`（例如 `tenant -> resource`）。
2. 支持父子双向互斥：`L1` 与该 `L1` 下任意 `L2` 不可并存（持有 `L1` 时阻塞新 `L2`；存在任意 `L2` 时 `L1` 必须等待）。
3. 提供可重试 API：`acquire/release` 通过 `request_id` 实现幂等。
4. 向调用方提供可观察失效语义：`LockHandle.Done()` 事件通知。
5. 提供 fencing token，帮助业务侧实现“新写覆盖旧写”的防护。

### 1.2 安全目标（Fail-Closed）

1. 客户端侧先失效：本地 TTL + heartbeat 失败策略优先让锁在业务侧不可用。
2. 服务端侧兜底回收：session lease 到期后服务端最终回收锁。

### 1.3 非目标

1. 不是多副本强一致分布式锁。
2. 不是跨机房容灾锁。
3. 不提供持久化锁状态恢复。

## 2. 架构与关键机制

### 2.1 组件

1. `pkg/hierlock`：进程内层级锁核心。
2. `server`：单节点 gRPC 锁服务，管理 session、token、幂等缓存。
3. `client`：Go SDK，管理 session 生命周期、watch/heartbeat、本地 TTL。

### 2.2 核心对象

1. Session：服务端租约单位，锁隶属于 session。
2. Lock token：每次 acquire 返回的锁标识。
3. Fence：全局单调递增计数（进程内），供下游写入校验。
4. Request ID：客户端重试幂等键（session 内生效）。

### 2.3 关键并发策略

1. `hierlock` 中 `L1` 用 `RWMutex`：
   - `LockL1` 走写锁。
   - `Lock(L1,L2)` 先拿 `L1` 读锁，再拿 `L2` 写锁。
   - 因为 `LockL1` 只有在没有任何 `L1` 读锁持有者时才会成功，所以 `LockL1` 成功返回时该前缀下不存在在持有中的 `L2`。
2. `pendingWriters` 门控：
   - 有 `L1` 写者等待时，阻止新 `L1` 读者进入，缓解写者饥饿。
3. 自旋 + 退避：
   - 使用 `TryLock/TryRLock` + `runtime.Gosched` + 指数退避睡眠。
4. Session 过期计时：
   - 服务端使用 `timerSeq` 防止旧定时器回调误删新 lease。

## 3. 实现权衡

### 3.1 为什么是“单节点 + 会话租约”

优点：

1. 结构简单，延迟低，便于排障。
2. 一致性模型清晰，不依赖外部存储。

代价：

1. 单点故障。
2. 重启后内存态丢失（包括锁、session、幂等缓存、fence 计数）。

### 3.2 为什么是“客户端先失效、服务端后回收”

优点：

1. 业务侧异常写入窗口更短。
2. 网络抖动时，倾向保守停写，降低并发破坏风险。

代价：

1. 可用性会被安全策略牺牲（例如 heartbeat 连续失败两次即 fail-closed）。
2. 需要调用方认真处理 `Done()` 事件。

### 3.3 为什么幂等是“容量+TTL 双有界窗口”

优点：

1. 限制内存增长，避免 session 长时间运行导致无限膨胀。
2. 通过“最旧淘汰 + TTL 过期”双机制，降低重试风暴导致的长期占用。

代价：

1. 超出容量或超过 TTL 后，老 `request_id` 重试不再幂等。
2. 调用方需要在合理时间窗口内完成重试。

## 4. 逐场景行为分析

### 场景 A：同一 `L1/L2` 并发抢锁

预期：

1. 同键互斥，仅一个持有者成功。
2. 其他请求阻塞到超时或前一个释放。

结论：当前实现符合预期。

### 场景 B：同一 `L1` 下不同 `L2`

预期：可并行获取，不互斥。

结论：当前实现符合预期，依赖 `L2` 独立节点锁。

### 场景 C：`L1` 写锁与 `L2` 读者竞争

预期：

1. 已持有的 `L2` 可以执行完。
2. 新 `L2` 进入被 `pendingWriters` 门控阻止。
3. 写者最终可获得 `L1`。

结论：当前实现符合预期，写者饥饿显著缓解。

### 场景 D：客户端超时/取消

预期：`context` 到期后快速失败，不产生悬挂锁。

结论：当前实现符合预期，失败路径均有回滚。

### 场景 D2：Lock 结果不确定（网络抖动/响应丢失）

预期：

1. 若服务端未明确返回 `LOCK_TIMEOUT`，客户端将该失败视为“不确定是否已加锁”。
2. 客户端自动 fail-closed 当前 session，广播 `LOCK_UNCERTAIN_RESULT`，并异步关闭远端 session。

结论：当前实现已按保守策略处理，优先避免挂锁长期占用。

### 场景 E：客户端异常退出（未主动 unlock）

预期：服务端在 lease 到期后回收锁。

结论：符合预期。注意这是“最终回收”，不是即时回收。

### 场景 F：heartbeat 瞬时失败

预期：单次失败不立即失效，连续失败后 fail-closed。

结论：符合当前策略。该策略偏保守，牺牲部分可用性换安全性。

### 场景 G：服务端重启/实例切换

预期：

1. 优雅重启时，服务端通过 `WatchSession` 广播 `SESSION_INVALIDATED(server_restarting)`。
2. 本地 session 失效，锁事件通知 `session_gone`。
3. 非优雅中断时，客户端通过 stream 断开触发 fail-closed。

结论：符合预期。锁状态不会跨重启保留。

### 场景 H：重试风暴 + 大量 `request_id`

预期：幂等缓存有界，不无限涨内存。

结论：当前为“容量 + TTL 双有界幂等缓存”。超过容量或 TTL 的老请求不保证幂等。

### 场景 I：高竞争热点 key

现状：

1. 存在自旋 + 退避行为。
2. 能控制阻塞，但在极端热点下会增加 CPU 消耗与尾延迟。

结论：可用但需压测验证容量。

## 5. 当前已知问题与风险

### 5.1 架构级限制（重要）

1. 单节点单点故障。
2. 内存态模型，重启即丢锁态与幂等态。
3. `fence` 为进程内单调，重启后从头计数；如果业务侧仅按数字比较且不区分 `server_id`，可能误判新旧。

### 5.2 工程级风险

1. 客户端若未调用 `Close(ctx)`，后台 goroutine（heartbeat/sweep）会泄漏。
2. 默认鉴权为 Bearer token（`LOCK_SERVER_AUTH_TOKEN`）；不可信网络建议进一步配合 mTLS。
3. 已有内建拦截器日志与指标快照；仍建议补充 Prometheus 与可视化看板。

### 5.3 行为语义边界

1. `request_id` 幂等仅在“同 session + 窗口内”成立。
2. 客户端 `Done()` 是本地视角失效通知，不等同于分布式全局线性化事件。
3. `LockOption.LocalTTL` 为客户端本地句柄语义，服务端不可见，也不会改变服务端 session lease。
4. `SessionLease` 是客户端请求值，不是服务端静态配置项；最终值受服务端策略约束（默认值/上限）。
5. 客户端崩溃且未主动 `Unlock/Close` 时，锁释放依赖服务端 session lease 到期回收。

## 6. 使用注意事项（强约束）

1. `Client` 必须复用，不要每次请求新建一个。
2. 每个 `Client` 结束时必须调用 `Close(ctx)`。
3. 必须消费 `LockHandle.Done()`，并在 `session_gone/server_changed/local_expired` 时立即停止临界区操作。
4. `request_id` 必须全局高熵唯一（至少在 session 内唯一），重试时复用同一个 `request_id`。
5. 业务落库必须使用 fencing token 进行比较（例如 `WHERE fence < newFence`）。
6. `LocalTTL` 应小于服务端 lease（或使用默认推导），确保客户端先失效。
7. 生产环境至少启用 `LOCK_SERVER_AUTH_TOKEN`，并建议配合 mTLS 或网关鉴权。

## 7. 参数建议

1. `HeartbeatInterval`：100ms 到 1000ms（按业务写入敏感度选择）。
2. `HeartbeatTimeout`：通常设置为 `2x~3x` 的 p99 RTT。
3. `LocalTTL`：建议 `2x~4x HeartbeatInterval`。
4. 服务端 lease：建议 `LocalTTL + buffer`，`buffer` 至少覆盖一次 heartbeat 抖动。
5. `LOCK_SERVER_SHARDS`：按热点分布压测后调整，常见从 `256/512/1024` 起步。

## 8. 运维与测试建议

### 8.1 必做压测场景

1. 热点单 key（同一 `L1/L2`）高并发争抢。
2. 热点前缀（同一 `L1` 不同 `L2`）混合流量。
3. 心跳抖动（延迟突增、超时、短时 5xx）。
4. 服务重启 + 客户端重连风暴。
5. 限流边界（RPS/Burst）与拒绝率曲线。

### 8.2 故障演练

1. kill -9 服务端，验证业务是否按 `Done()` 停写。
2. 注入网络抖动，验证是否发生错误持锁写入。
3. 幂等窗口边界测试，验证重试策略不会依赖“无限幂等”。

## 9. 后续演进方向

1. 引入可持久化的 fence epoch（或把 `server_id` 纳入 fence 语义）。
2. 增加观测面：Prometheus 指标、结构化日志、trace。
3. 支持多实例时引入一致性存储（etcd/raft）或明确网关粘性与单主约束。
4. 客户端提供显式 `Shutdown`/`Close` 语义文档与 lint 规则，降低资源泄漏误用概率。
